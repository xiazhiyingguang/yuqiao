# 语桥 Flutter iOS UI 与开发指南

> 适用对象：没有 Mac、没有 iPhone，但准备使用 Windows + VS Code + Flutter 开发苹果风格移动应用原型的参赛团队。  
> 项目方向：语桥——面向中文轻中度失语患者的找词与表达辅助 App。

---

## 0. 结论与推荐路线

你的现实条件是：

- 没有苹果手机；
- 没有 Mac；
- 准备使用 Flutter；
- 需要做一个符合苹果风格的手机端 App；
- 最终最好能在 iOS 设备或云真机上展示。

推荐路线：

```text
Windows + VS Code + Flutter
↓
Android / Web 进行日常预览和调试
↓
使用 Flutter 做出 iOS 风格 UI
↓
使用 Codemagic / GitHub Actions macOS Runner 云端构建 iOS .ipa
↓
上传云真机，或借 iPhone 通过 TestFlight / Ad Hoc 测试
↓
录制演示视频 / 参加答辩
```

核心判断：

> Flutter 可以让你在 Windows 上完成大部分开发，但 iOS 编译、签名、真机测试需要云构建或借助苹果设备完成。

---

## 1. 产品 UI 总目标

语桥不是娱乐 App，也不是普通聊天 App，它是一个面向轻中度失语患者的表达辅助工具。

UI 应该传达四个关键词：

1. **清楚**：用户一眼知道下一步点哪里；
2. **温和**：不要让患者感到被治疗、被监控或被标签化；
3. **可靠**：界面像工具，不像玩具；
4. **低负担**：减少文字、减少选择、减少跳转。

建议整体风格参考：

- Apple Health 的健康可信感；
- Reminders 的简洁列表；
- Shortcuts 的流程感；
- Settings 的分组表单；
- Home 的大按钮控制卡片。

不要参考：

- 花哨大屏后台；
- 颜色很多的儿童 App；
- 密密麻麻的功能宫格；
- 重阴影、重渐变、重装饰的视觉风格。

## 2. 苹果风格 UI 的核心原则

### 2.1 原生感优先

即使用 Flutter，也要尽量模仿 iOS 的交互结构：

- 底部 Tab；
- 大标题导航；
- 分组列表；
- 底部弹出 Sheet；
- 大圆角卡片；
- 系统级留白；
- 轻量反馈动画。

Flutter 中可以优先使用：

```dart
CupertinoApp
CupertinoPageScaffold
CupertinoNavigationBar
CupertinoTabScaffold
CupertinoButton
CupertinoListSection
CupertinoListTile
CupertinoActionSheet
```

但实际开发中，不必完全使用 Cupertino 组件。你也可以使用普通 `Container`、`ListView`、`GridView` 自定义出 iOS 风格。关键是视觉和交互要克制。

---

### 2.2 信息层级清楚

每个页面只回答一个问题。

错误示例：

```text
一个页面同时出现：拍照、词库、AI、历史、设置、家属管理、训练记录。
```

正确示例：

```text
页面标题：你想表达哪一类？
选项：
- 我想要……
- 我觉得……
- 我不舒服……
- 我想问……
```

对语桥来说，每一屏最多 3–4 个主要选择。不要让失语患者在一页里看到十几个按钮。

---

### 2.3 少颜色，多留白

苹果风格不是“每个按钮一个颜色”。

推荐：

| 用途 | 建议 |
|---|---|
| 页面背景 | 极浅灰 `#F5F5F7` |
| 卡片背景 | 白色 `#FFFFFF` |
| 主色 | 柔和蓝 `#3478F6` |
| 次级背景 | 浅蓝灰 `#EEF3FA` |
| 正文文字 | 深灰黑 `#1C1C1E` |
| 辅助文字 | 中灰 `#8E8E93` |
| 危险提醒 | 系统红 `#FF3B30` |
| 成功状态 | 系统绿 `#34C759` |

颜色使用建议：

- 主按钮用蓝色；
- 普通候选卡片用白色或浅灰；
- 医疗/疼痛/紧急表达才使用红色或橙色；
- 不要大面积渐变；
- 不要使用高饱和荧光色。

---

### 2.4 字体要大，且支持无障碍

语桥的目标用户可能存在阅读负担，所以字体要比普通 App 大一些。

建议字号：

| 场景 | 字号 |
|---|---:|
| 页面大标题 | 30–34 |
| 页面说明 | 17–20 |
| 候选按钮文字 | 22–26 |
| 主按钮文字 | 20–24 |
| 辅助说明 | 14–16 |

Flutter 里可以定义统一文字样式：

```dart
class AppTextStyles {
  static const title = TextStyle(
    fontSize: 32,
    fontWeight: FontWeight.w700,
    height: 1.2,
  );

  static const sectionTitle = TextStyle(
    fontSize: 22,
    fontWeight: FontWeight.w700,
  );

  static const body = TextStyle(
    fontSize: 18,
    fontWeight: FontWeight.w400,
    height: 1.4,
  );

  static const candidate = TextStyle(
    fontSize: 24,
    fontWeight: FontWeight.w700,
  );
}
```

注意：不要把所有字号写死到很小。后续可以通过系统字体缩放、老人模式或 App 内设置进行调整。

---

### 2.5 候选数量要少

语桥最重要的不是给用户“更多选择”，而是给用户“更容易确认的选择”。

推荐规则：

```text
普通候选：3–4 个
AI 候选句：2–3 个
首页主入口：2 个最大按钮 + 4 个常用入口
每一步都保留：返回 / 都不是 / 重新选择
```

## 3. 语桥的信息架构设计

建议 App 只保留 3 个底部 Tab：

```text
表达    词库    我的
```

### 3.1 表达 Tab

这是主页面。

包含：

- 我卡住了；
- 拍照找词；
- 常用表达；
- 最近表达。

页面示意：

```text
语桥
说不出时，先从这里开始

[ 我卡住了 ]
[ 拍照找词 ]

常用表达
[ 我想问医生 ] [ 我不舒服 ]
[ 请再说一遍 ] [ 我想联系家人 ]

最近表达
- 我最近右手发麻，晚上更明显
- 我想问医生一个问题
```

### 3.2 词库 Tab

这是家属和患者共同维护的个人表达记忆库。

包含：

- 人物；
- 身体；
- 感觉；
- 地点；
- 物品；
- 药品；
- 常用句。

页面示意：

```text
词库

人物
妈妈、女儿、医生、康复师

身体
头、手、腿、肚子、胸口

感觉
疼、麻、晕、累、冷、热

药品
我的降压药、止痛药、胰岛素
```

### 3.3 我的 Tab

这是设置与管理区域。

包含：

- 患者档案；
- 家属管理；
- 常用语管理；
- 表达记录；
- 隐私与授权；
- 字体与按钮大小；
- 关于语桥。

## 4. 核心页面设计

### 4.1 首页

首页最重要的是两个主按钮：

1. **我卡住了**；
2. **拍照找词**。

“我卡住了”应该最大，因为这是你的核心流程。

设计建议：

- 顶部大标题：语桥；
- 副标题：说不出时，先从这里开始；
- 两个大卡片按钮；
- 常用表达放在下方；
- 最近表达放在更下方。

---

### 4.2 “我卡住了”页面

目标：帮助用户选择表达类型。

页面文案：

```text
你想表达哪一类？

[ 我想要…… ]
[ 我觉得…… ]
[ 我不舒服…… ]
[ 我想问…… ]
```

设计原则：

- 只给 4 个选择；
- 每个选择配一个图标；
- 按钮足够大；
- 底部有“返回首页”。

---

### 4.3 候选词页面

示例：用户选择“我不舒服”。

第一步：

```text
哪里不舒服？

[ 头 ]    [ 右手 ]
[ 胸口 ]  [ 肚子 ]

[ 都不是 ]
```

第二步：

```text
是什么感觉？

[ 疼 ]    [ 麻 ]
[ 晕 ]    [ 恶心 ]

[ 都不是 ]
```

第三步：

```text
什么时候更明显？

[ 早上 ]      [ 晚上 ]
[ 现在 ]      [ 一直 ]

[ 跳过 ]
```

---

### 4.4 AI 候选句页面

这是语桥的核心创新展示页。

页面文案：

```text
请选择最接近的一句

[ 我右手发麻。 ]
[ 我最近右手发麻，晚上更明显。 ]
[ 我想问医生，右手发麻是不是需要注意。 ]

[ 都不对，重新选择 ]
```

设计重点：

- 候选句使用大卡片；
- 每张卡片只放一句话；
- 不要生成太长内容；
- 点击某句后进入确认页；
- 一定要有“都不对”。

---

### 4.5 播报确认页面

页面文案：

```text
我想说：

我最近右手发麻，晚上更明显。

[ 播报给对方 ]
[ 换一种说法 ]
[ 保存为常用语 ]
```

设计重点：

- 主按钮放底部；
- 主按钮要大；
- 播报前必须确认；
- 医疗相关内容可以使用二次确认。

## 7. iOS 风格开发注意事项

### 7.1 不要默认做成 Android 风格

Flutter 默认很多教程会使用 Material 组件，这不代表你的 App 必须像 Android。

你要避免：

- 默认蓝色 AppBar；
- 右下角悬浮按钮；
- 强阴影 Card；
- 密集表单；
- Drawer 侧边栏；
- 过多 Material 动效。

### 7.2 要模拟 iOS 的页面感觉

建议：

- 背景浅灰；
- 卡片白色；
- 圆角 20–24；
- 顶部大标题；
- 底部 Tab；
- 主要操作按钮放底部；
- 返回按钮清楚；
- 页面层级不超过 3 层。

### 7.3 安全区适配

iPhone 有刘海、灵动岛、底部 Home Indicator。Flutter 里尽量使用：

```dart
SafeArea(
  child: ...
)
```

底部按钮要留足空间：

```dart
padding: const EdgeInsets.fromLTRB(20, 16, 20, 28)
```

### 7.4 深色模式

第一版可以不强制做，但最好使用统一颜色管理，后续方便适配。

不要在各处散落写死颜色。

## 8. 无障碍设计清单

语桥的无障碍不是附加功能，而是核心竞争力。

必须做到：

1. 按钮足够大；
2. 文字足够大；
3. 每屏选项不超过 4 个；
4. 有返回、撤销、都不是；
5. 播报前必须确认；
6. 颜色不能作为唯一信息；
7. 医疗表达要更谨慎；
8. 支持语音反馈；
9. 避免复杂手势；
10. 避免长段文字。

建议增加一个“易用模式”：

```text
易用模式开启后：
- 字号增大；
- 每屏只显示 2–4 个选项；
- 按钮高度增加；
- 减少动画；
- 所有候选可点击朗读预览。
```

## 10. iOS 构建方案

### 10.1 你为什么还需要 iOS 构建

Flutter 是跨平台框架，但 iOS App 最终仍然需要被编译成 iOS 安装包。

iOS 安装包通常是 `.ipa` 文件。

没有 Mac 时，你不能在本地生成 `.ipa`，但可以使用云端 macOS 构建服务。

推荐选择：

- Codemagic；
- GitHub Actions macOS Runner；
- Bitrise；
- 云 Mac。

## 13. 推荐核心 Demo

不要展示太多功能，展示一个完整故事。

场景：

> 轻中度失语患者复诊时，想表达“我最近右手发麻，晚上更明显，想问医生是不是需要注意”。

演示流程：

```text
打开语桥
↓
点击“我卡住了”
↓

↓
选择“身体不舒服”
↓
选择“右手”

选择“麻”
↓
选择“晚上更明显”
↓
App 生成候选句
↓
用户确认
↓
手机语音播报
↓
保存为医疗常用语
```

这条线最能体现：

- 找词辅助；
- 句子组织；
- AI 候选；
- 用户确认；
- 语音输出；
- 医疗场景价值。

## 14. 不建议第一版做的功能

第一版不要做太大。

暂缓功能：

- 自动卡顿检测；
- 全天候对话监听；
- 眼动识别；
- 音色复刻；
- 完整拍照识物；
- 医疗级康复评估；
- 医生后台；
- 多端同步。

原因：

- 开发成本高；
- 测试难；
- 风险重；
- 容易分散主线；
- 比赛展示不一定加分。

MVP 只聚焦：

```text
找词 → 补词 → 成句 → 确认 → 播报
```

## 15. iOS 权限注意事项

如果后续加入拍照、语音识别、麦克风，需要在 iOS 配置中写权限说明。

常见权限：

```text
相机权限：用于拍照识别物品，帮助用户找词。
麦克风权限：用于语音输入或卡顿辅助。
语音识别权限：用于将用户语音转成文字。
照片权限：用于添加家属照片和个人物品图片。
```

权限文案要清楚、克制，不要写得像监控用户。

推荐文案：

```text
语桥需要使用相机，以便在您主动拍照时识别物品名称。
语桥需要使用麦克风，以便在您主动开启时辅助识别语音内容。
语桥需要访问照片，以便添加家人、物品或药品图片。
```

展示时不要说：

答辩重点：

2. 目标人群明确；
3. 核心闭环完整；
4. UI 低负担；
5. AI 不替用户决定；
6. 技术可落地；
7. Flutter 可跨平台，后续可通过云构建适配 iOS。

## 18. 最终建议

语桥的核心竞争力不是“做了多少功能”，而是一个清楚、可靠、低负担的表达闭环：

```text
我卡住了
↓
选择表达类型
↓
选择候选词
↓
AI 生成候选句
↓
患者确认
↓
语音播报
```

Flutter 可以帮你在 Windows 上完成大部分开发。你需要接受的限制是：iOS 最终构建和真机测试需要云服务或借助苹果设备。

所以你的开发策略应该是：

> 先把 App 做得可用、好看、流程完整；再解决 iOS 打包和真机测试。
