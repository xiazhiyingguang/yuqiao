# 《语桥》项目计划书

**面向轻中度失语人群的情境感知式 AI 辅助沟通 App**

---

## 一、项目名称

**语桥：面向轻中度失语人群的情境感知式 AI 辅助沟通 App**

副标题：
基于多模态识别、上下文预测与确认式生成的中文 AAC 表达辅助系统

项目口号：
**说不出时，语桥帮你找回想说的话。**

---

## 二、项目背景

失语症是脑卒中、脑外伤、脑肿瘤、神经退行性疾病等导致的常见语言障碍。部分患者意识清醒、思维完整，但在真实交流中会出现"想说但说不出""知道眼前物品用途但叫不出名称""说话过程中突然卡住""只能表达零散词语""无法组织完整句子"等问题。

对于轻中度失语患者而言，困难往往并不是完全没有表达意图，而是缺少合适的表达路径。患者可能知道自己想表达什么，却难以找到准确的词语、合适的句式，或者无法把零散关键词组织成他人能够理解的完整句子。

传统 AAC，即辅助与替代沟通工具，能够帮助语言障碍人群完成基本表达，但现有方案普遍存在以下问题：

1. **输入负担较高**：患者需要在大量词库中主动寻找目标词，对找词困难患者并不友好。
2. **表达启动困难**：许多系统提供静态词库，但无法帮助患者从当前情境中找到表达入口。
3. **表达中断支持不足**：患者说到一半卡住时，传统工具难以根据上下文继续补词。
4. **中文场景适配不足**：现有产品多以英文或通用符号系统为主，缺少中文家庭照护、医患沟通、亲情表达等语境。
5. **个性化不足**：患者的家人、常用物品、药品、生活习惯等信息难以被充分记忆和复用。
6. **AI 表达边界不清**：若系统直接替患者生成并播报内容，可能造成误表达，损害患者表达自主权。

因此，本项目提出"语桥"App，面向轻中度失语人群，通过高频表达、拍照识物、对话补词、图文化候选、AI 句子整理、二次确认和语音播报，帮助患者在真实交流场景中降低表达成本，重新建立与家属、医护人员和社会环境之间的沟通桥梁。

## 三、目标用户

### 3.1 核心用户

本项目主要面向以下人群：

1. **命名性失语患者**
   患者知道眼前物品是什么、用途是什么，但无法叫出其名称。例如看到杯子却说不出"杯子"，看到药盒却叫不出"药"。

2. **轻中度表达性失语患者**
   患者理解能力相对保留，但语言表达受限，常出现找词困难、表达中断、语句组织困难。

3. **脑卒中后语言障碍患者**
   尤其是意识清醒、理解能力相对保留、手部功能基本完好、能够进行简单点击操作的患者。

4. **渐进性语言障碍或运动障碍患者**
   如部分 ALS、原发性进行性失语等患者，可在疾病早期建立个性化词库和声音档案。

### 3.2 辅助用户

1. 患者家属；
2. 护理人员；
3. 康复治疗师；
4. 医护人员；
5. 社区养老与康复机构工作人员。

### 3.3 非核心用户边界

本项目第一阶段不主要面向：

1. 意识不清、无法完成确认操作的患者；
2. 严重理解障碍患者；
3. 严重视觉障碍患者；
4. 无法进行稳定点击且暂未接入眼控、开关控制的重度运动障碍患者；
5. 需要医学诊断、康复评估或治疗建议的临床场景。

本项目定位为辅助沟通工具，不替代医生诊断、康复治疗或专业医疗设备。

## 四、项目定位

"语桥"不是让 AI 替患者说话，而是通过 AI 辅助患者完成表达。

项目核心原则是：

**AI 提供候选，患者确认含义，系统整理表达。**

项目重点解决轻中度失语患者在真实交流中遇到的两类核心困难：

### 4.1 表达启动困难

表达启动困难指患者有表达需求，但不知道如何开始表达。患者可能想喝水、想见家人、想说身体不舒服，或者想指出眼前某个物品，却无法找到合适的词语或句式。

语桥针对该问题提供三类表达入口：

1. **高频表达首页**
   直接提供"我要喝水""我要上厕所""我疼""叫医生""我想见家人"等常用表达。

2. **拍照识物入口**
   当患者看见某个物品却叫不出名称时，可以拍照识别。系统不仅返回物品名，还推荐相关表达。

3. **句式骨架入口**
   提供"我想__""我需要__""我觉得__""请帮我__"等表达模板，帮助患者从句式结构开始表达。

表达启动链路如下：

> 患者有表达需求，但不知道如何开始
> ↓
> 进入高频表达首页 / 拍照识物 / 选择句式骨架
> ↓
> 系统结合时间、地点、拍摄物体、个人词典和历史表达习惯进行情境感知
> ↓
> AI 推荐 3–4 个表达候选
> ↓
> 患者点击确认
> ↓
> AI 整理为完整句子
> ↓
> 患者二次确认
> ↓
> 系统语音播报

### 4.2 表达中断困难

表达中断困难指患者已经开始表达，但说到一半突然卡住。比如患者说出"我想……""那个东西……""帮我……"，却无法继续说出后面的词语。

语桥针对该问题设计"对话模式"和"我卡住了"辅助机制。用户在对话开始前主动开启对话模式，系统在授权范围内记录当前对话上下文。当患者出现明显卡顿，或者主动点击"我卡住了"时，系统根据已有表达内容、当前场景、个人词典和历史习惯推荐可能的下一个词或短语。

表达中断链路如下：

> 用户主动开启"对话模式"
> ↓
> 系统记录当前对话内容并建立上下文
> ↓
> 结合时间、地点、近期表达记录及个人词典持续更新语境模型
> ↓
> 患者出现明显卡顿或主动点击"我卡住了"
> ↓
> AI 根据此前已表达内容、当前语境、多模态信息和历史习惯推荐下一个词或短语
> ↓
> 患者点击确认
> ↓
> AI 将零散词语整理为清晰句子
> ↓
> 患者二次确认
> ↓
> 系统语音播报

其中，"我卡住了"不是独立于对话之外的孤立功能，而是对话模式中的辅助功能。只有在系统已经获取当前对话上下文，并结合用户历史习惯和环境信息的前提下，才能更准确地预测患者想表达的内容。

### 4.3 个性化自适应学习

语桥需要在用户授权前提下持续学习患者的表达习惯。系统记录高频词汇、常用句式、常见人物、药品、地点、护理偏好等信息，并动态调整候选词排序权重。

家属或言语治疗师也可以补充患者专属词典，例如亲友姓名、宠物名、常用地址、常用药品、常见症状、护理偏好等，使个性化词汇自动融入预测模型。

个性化学习链路如下：

> 用户授权个性化学习
> ↓
> 系统记录高频词汇、常用句式、人物、药品、地点等信息
> ↓
> 家属或治疗师补充患者专属词典
> ↓
> 系统动态调整候选词排序权重
> ↓
> 个性化词汇融入预测模型
> ↓
> 实现"越用越懂你"的表达辅助体验

### 4.4 环境与多模态感知预测

语桥综合利用移动设备传感器和多模态输入信息，生成更符合当前真实语境的候选表达。

主要包括：

1. **时间感知**
   早晨优先推荐"早餐""吃药""喝水"；晚上优先推荐"休息""翻身""关灯"。

2. **位置感知**
   在医院优先推荐"医生""检查""疼痛""药"；在餐厅优先推荐食物和饮料。

3. **视觉感知**
   通过拍照识物识别当前场景中的关键物体，并将识别结果转化为可表达内容。

4. **对话上下文感知**
   在对话模式中，根据患者已经说出的内容预测后续表达。

5. **个人习惯感知**
   结合历史表达记录修正预测结果，使候选表达越来越贴近具体患者。

### 4.5 项目本质

两条核心链路共享情境感知、个性化学习、候选推荐、句子整理和确认播报能力，但分别解决"不会开始说"和"说到一半说不下去"两种不同表达障碍。

项目的本质不是"AI 识别患者真实意图"，而是：

**通过情境感知、多模态理解、个性化学习、候选推荐和确认式生成，帮助患者用更少操作表达自己。**

## 五、核心功能概览

根据项目定位，语桥的核心功能不应简单理解为若干独立工具的组合，而应围绕轻中度失语患者在真实交流中的两类主要困难展开：一类是启动困难，即患者有表达意图但不知道如何开始说；另一类是表达中断，即患者已经开始表达，但说到一半突然找不到词、组织不了句子。

因此，语桥的核心功能体系可以概括为：

**表达入口层 + 情境感知层 + 候选推荐层 + 确认生成层 + 个性化学习层。**

### 5.1 表达启动辅助

**功能定位**

表达启动辅助主要解决患者"不知道如何开始表达"的问题。患者可能有明确需求，例如想喝水、想见家人、身体疼痛、想询问某个物品，但由于找词困难或组织语言困难，无法顺利开始表达。

语桥通过高频表达首页、拍照识物和句式骨架，为患者提供多个低负担表达起点。

**核心功能**

1. **高频表达首页**
   首页展示患者最常用的表达入口，例如"我要喝水""我要上厕所""我疼""我想休息""叫医生""我想见家人"等。患者可以通过一次点击快速发起表达。

2. **拍照识物入口**
   当患者看到眼前物品但想不起名称时，可以拍照识别。系统不仅返回物品名称，还给出与该物品相关的常用表达。例如识别到"药盒"后推荐"吃药""忘了吃药""药吃完了""帮我拿药"。

3. **句式骨架入口**
   系统提供"我想____""我需要__""我觉得__""请帮我__""我不想__"等句式模板，帮助患者从语法结构开始表达，而不是完全从词库中搜索。

**解决的问题**

1. 降低患者表达起步难度；
2. 避免患者在大量词库中盲目查找；
3. 帮助患者从场景、物品或句式三个方向发起表达；
4. 提升日常交流的启动效率。

**技术实现预测**

| 功能 | 技术实现 |
|------|----------|
| 高频表达首页 | SwiftUI / UIKit + 本地词库 |
| 高频词排序 | 本地使用频率统计 + 时间段权重 |
| 拍照识物 | AVFoundation + Vision / Core ML / 多模态模型 |
| 物品相关表达生成 | 规则模板 + LLM |
| 句式骨架 | 本地 JSON / SQLite 模板库 |
| 首页个性化 | 本地用户画像 + 历史表达记录 |

### 5.2 对话模式与表达中断辅助

**功能定位**

表达中断辅助主要解决患者"说到一半突然卡住"的问题。患者可能已经说出部分内容，例如"我想……""那个东西……""帮我……"，但无法继续说出后面的词语。

语桥通过对话模式记录上下文，并在患者主动点击"我卡住了"或系统检测到明显卡顿时，推荐可能的下一个词、短语或句式。

**核心功能**

1. **对话模式开关**
   患者在开始交流前主动开启对话模式。系统只在该模式下记录当前对话上下文，避免默认后台监听造成隐私问题。

2. **"我卡住了"主动触发**
   患者说不下去时，可以点击"我卡住了"按钮。系统根据已有对话内容、当前场景、历史表达习惯和个人词典推荐候选内容。

3. **辅助卡顿检测**
   在对话模式下，系统可以结合停顿时长和句子完整性判断是否可能发生表达中断。例如，当患者停顿超过 2–3 秒，且当前语句明显未完成时，系统弹出候选词。

4. **上下文续写候选**
   系统根据患者已经表达的内容推荐下一个词或短语。例如患者说出"我想……"，系统推荐"喝水""休息""见家人""上厕所"；患者选择"见家人"后，系统继续推荐"妈妈""爸爸""女儿""医生"等。

**解决的问题**

1. 帮助患者在表达中断时继续说下去；
2. 减少患者重新组织整句话的压力；
3. 让 AI 辅助建立在当前对话上下文之上；
4. 通过主动触发与辅助检测降低误打扰风险。

**技术实现预测**

| 功能 | 技术实现 |
|------|----------|
| 对话模式开关 | 前端显式按钮 + 权限提示 |
| 语音输入 | AVAudioEngine / iOS Speech Framework |
| 语音转文字 | Apple Speech / Whisper / 云端 ASR |
| 停顿检测 | 静音检测 + 时间阈值 |
| 句子完整性判断 | 规则判断 + LLM / 小模型 |
| 主动卡顿按钮 | SwiftUI 悬浮按钮 |
| 上下文推荐 | 当前语句 + 历史表达 + 个人词典 + LLM |

### 5.3 情境感知与多模态预测

**功能定位**

情境感知是语桥区别于普通 AAC 沟通板的关键能力。传统沟通板主要依赖静态词库，患者需要主动寻找词汇；语桥则根据时间、地点、拍照物体、当前对话和个人习惯，主动生成更接近真实语境的候选表达。

**核心功能**

1. **时间感知**
   系统根据时间段调整候选词权重。例如早晨优先推荐"早餐""吃药""喝水"，晚上优先推荐"休息""翻身""关灯"。

2. **位置感知**
   在用户授权前提下，系统根据地点优化候选。例如在医院优先推荐"医生""检查""疼痛""药"，在餐厅优先推荐"吃饭""喝水""点菜"。

3. **视觉感知**
   通过拍照识物识别当前环境中的关键物体，并将物体名称转化为可表达内容。例如识别到"杯子"后，不只显示"杯子"，还推荐"我要喝水""帮我拿杯子"。

4. **对话上下文感知**
   在对话模式中，系统根据前文内容预测患者下一步可能表达的词语或短语。

5. **个人习惯感知**
   系统结合患者历史表达记录和家属配置的个人词典，调整候选排序，使推荐结果越来越贴合具体患者。

**解决的问题**

1. 让候选词从"通用推荐"变成"当前场景推荐"；
2. 减少患者点击次数；
3. 提高表达候选的命中率；
4. 形成"越用越懂你"的产品体验。

**技术实现预测**

| 功能 | 技术实现 |
|------|----------|
| 时间感知 | 系统时间 + 规则权重 |
| 位置感知 | Core Location，可选授权 |
| 视觉感知 | Vision / Core ML / 多模态模型 |
| 上下文感知 | ASR 文本 + 当前选择状态 |
| 个人习惯感知 | 本地表达日志 + 高频统计 |
| 综合排序 | 规则引擎 + LLM 排序 / 重排 |

### 5.4 图文化候选推荐

**功能定位**

候选推荐是语桥的核心交互环节。AI 不直接替患者表达，而是根据情境生成 3–4 个候选词、短语或句式，让患者点击确认。

候选内容必须符合失语患者的认知特点：数量少、按钮大、图文结合、表达明确。

**核心功能**

1. **具体名词候选**
   对物品、人物、地点等具体内容，优先使用图片、图标或真实照片。例如"妈妈"使用家属照片，"降压药"使用药盒照片。

2. **动作与需求候选**
   对"喝水""吃药""翻身""休息"等动作表达，使用图标 + 大字标签。

3. **情绪与状态候选**
   对"难受""害怕""开心""累"等情绪状态，使用表情图标 + 大字标签。

4. **抽象表达候选**
   对"但是""不是""有点""想问"等难以图片化的内容，使用句式骨架承载逻辑关系，例如"我想_但是_""我不是_，我是_"。

5. **候选兜底按钮**
   每组候选都应提供"都不是"按钮，允许患者返回上一级、浏览分类词库或重新生成候选。

**解决的问题**

1. 降低阅读和理解负担；
2. 降低找词难度；
3. 避免候选过多造成认知压力；
4. 保护患者不被错误候选"卡死"。

**技术实现预测**

| 功能 | 技术实现 |
|------|----------|
| 候选生成 | 规则模板 + LLM |
| 候选排序 | 上下文相关性 + 个人使用频率 |
| 图标展示 | SF Symbols / 自定义 AAC 图标库 |
| 真实照片 | PhotosUI + 家属管理端 |
| 抽象句式 | 本地句式模板库 |
| 兜底路径 | 分类词库 + 搜索 / 拼音 / 手写输入 |

### 5.5 AI 句子整理与保守生成

**功能定位**

患者通过点击获得的内容往往是零散的，例如"我 / 肚子 / 疼 / 医生"。语桥需要将这些零散词语整理成清晰、自然、适合播报的句子。

但 AI 不能擅自替患者扩写含义，因此语句整理必须遵循"保守生成"原则：只整理患者已确认的信息，不添加未经确认的新内容；当含义存在歧义时，给出多个候选句。

**核心功能**

1. **关键词整理成句**
   将患者确认的关键词整理为完整句子。例如"我 / 肚子 / 疼 / 医生"整理为"我肚子很疼，请帮我叫医生。"

2. **多候选句生成**
   当关键词存在歧义时，系统不直接输出唯一句子，而是生成多个可能表达。例如"妈妈 / 疼 / 昨天"可以生成"我昨天很疼，想告诉妈妈""妈妈昨天很疼""我昨天和妈妈说我很疼"等候选。

3. **医疗表达保守处理**
   对疼痛、呼吸困难、叫医生等高风险表达，系统应尽量保持直接、明确、保守，不擅自进行医学解释。

4. **句子修改与重选**
   患者可以选择"不对，修改""重新选择""删除某个词""换一种说法"。

**解决的问题**

1. 帮助患者从零散表达过渡到完整表达；
2. 降低家属和医护人员理解成本；
3. 避免 AI 过度脑补患者意图；
4. 保护患者表达自主权。

**技术实现预测**

| 功能 | 技术实现 |
|------|----------|
| 零散词收集 | 本地状态管理 |
| 句子整理 | LLM API / 本地小模型 |
| 保守生成控制 | Prompt 约束 + 输出规则 |
| 多候选句生成 | LLM n-best 输出 |
| 医疗表达识别 | 关键词分类 + 风险分级 |
| 句子修改 | 前端编辑 + 重新生成 |

### 5.6 二次确认与语音播报

**功能定位**

语桥的最终输出不是 AI 自动播报，而是患者确认后的语音表达。因此，系统必须在"AI 整理句子"和"语音播报"之间加入二次确认机制。

**核心功能**

1. **句子确认页**
   系统将整理后的句子用大字展示，患者可以选择"播报""不对，修改""重新选择"。

2. **语音预览**
   对较长句子或重要表达，系统可先进行低干扰预览，帮助患者确认语句是否正确。

3. **风险分级确认**
   普通表达可以一次确认；医疗、疼痛、紧急求助类表达需要二次确认或长按确认。

4. **TTS 语音播报**
   患者确认后，系统使用语音合成将句子播报出来，完成面对面交流。

5. **个性化音色拓展**
   基础版使用系统 TTS；高级版可在明确授权前提下建立个人声音档案，实现更接近患者本人音色的播报。

**解决的问题**

1. 防止 AI 误表达；
2. 让患者拥有最终表达权；
3. 提高家属和医护人员对系统的信任；
4. 完成从候选到真实交流的闭环。

**技术实现预测**

| 功能 | 技术实现 |
|------|----------|
| 句子确认页 | SwiftUI Sheet / Modal |
| 大字展示 | Dynamic Type / 无障碍字体 |
| 语音预览 | AVSpeechSynthesizer |
| 语音播报 | AVSpeechSynthesizer / 云端 TTS |
| 风险分级 | 表达类型分类 + 规则判断 |
| 音色复刻 | 云端 TTS / 声音克隆服务，可选 |

### 5.7 个性化词典与家属管理

**功能定位**

语桥不是通用聊天工具，而是面向具体患者的长期辅助表达系统。因此，系统需要通过家属管理端建立患者专属词典，并将这些信息融入候选推荐。

**核心功能**

1. **亲友信息管理**
   家属可以添加亲友姓名、称呼和照片，例如妈妈、爸爸、女儿、孙子、护工、医生等。

2. **常用物品管理**
   家属可以添加患者常用物品，例如水杯、轮椅、药盒、遥控器、眼镜、助听器等。

3. **药品与护理信息管理**
   家属可以添加常用药、护理偏好、常见症状和紧急联系人。

4. **常用表达管理**

5. **个人物品识别记忆**
   系统将患者拍摄的物品与家属标注的个性化标签绑定，使后续识别更贴近真实生活。

**解决的问题**

1. 提升候选推荐准确度；
2. 增强长期使用价值；
3. 让 App 从通用工具变成患者专属表达助手；
4. 帮助患者表达更具体、更有情感温度的内容。

### 5.8 自适应学习与候选排序

**功能定位**

语桥需要在使用过程中逐渐学习患者的表达习惯，使系统推荐越来越准确。自适应学习不是替患者做决定，而是调整候选排序，让更可能被使用的表达更靠前。

**核心功能**

1. **高频表达学习**
   记录患者最常使用的词、短语和句式，并在首页或候选区优先展示。

2. **时间段习惯学习**
   根据不同时间段调整推荐内容。例如早晨优先推荐"喝水""吃药"，晚上优先推荐"休息""翻身"。

3. **场景习惯学习**
   根据医院、家庭、餐厅等不同场景调整表达候选。

4. **近期表达增强**
   如果患者最近多次表达"右腿疼"，系统在疼痛相关候选中提高"右腿疼"的优先级。

5. **可关闭学习**
   个性化学习必须在用户授权前提下进行，并支持关闭、清除和重置。

**解决的问题**

1. 降低重复表达成本；
2. 提高候选命中率；
3. 形成"越用越懂你"的产品体验；
4. 兼顾个性化与隐私保护。

**技术实现预测**

| 时间权重 | 时间段规则 |
| 场景权重 | 位置授权 + 使用场景标签 |
| 推荐排序 | 规则权重模型 |
| 学习开关 | 隐私设置页面 |
| 数据删除 | 本地清除 / 云端同步删除，可选 |

### 5.9 表达记录与照护辅助

**功能定位**

表达记录用于帮助家属和护理人员了解患者近期需求变化。它不是医疗诊断工具，而是照护沟通的辅助记录。

**核心功能**

### 5.10 隐私、安全与权限控制

**功能定位**

由于语桥涉及拍照、语音、位置、个人词典、表达记录和可选音色复刻，隐私和安全机制本身应作为核心功能设计，而不是附加说明。

**核心功能**

1. **显式模式开关**
   拍照识物、对话模式、位置感知、个性化学习都必须由用户或家属主动开启。

2. **本地优先处理**
   高频词库、表达记录、个性化词典等敏感数据优先存储在本地。

3. **云端调用提示**
   当需要调用云端多模态模型、大语言模型或音色服务时，系统应提示用户相关数据可能被上传处理。

4. **数据删除与重置**
   用户可以删除表达记录、个人词典、物品记忆和声音档案。

5. **音色复刻授权**
   音色复刻必须作为可选高级功能，需要患者或监护人明确授权，默认关闭。

**解决的问题**

1. 降低隐私争议；
2. 增强家属和患者信任；
3. 适应真实医疗和家庭照护场景；
4. 避免 AI 生成和声音复刻带来的伦理风险。

**技术实现预测**

| 功能 | 技术实现 |
|------|----------|
| 权限管理 | iOS Privacy Permission |
| 本地加密 | Keychain / 加密数据库 |
| 云端调用提示 | 前端弹窗 + 设置页 |

| 音色授权 | 授权协议 + 声音样本管理 |

### 5.12 核心功能总结

按照项目定位，语桥真正必须具备的核心能力不是"拍照识别""AI 聊天"或"语音播报"中的某一个单点，而是一套完整的辅助表达机制：

> 表达启动
> ↓
> 情境感知
> ↓
> 候选推荐
> ↓
> 患者确认
> ↓
> AI 整理
> ↓
> 二次确认
> ↓
> 语音播报
> ↓
> 个性化学习

其中，启动困难由"高频表达首页、拍照识物、句式骨架"解决；表达中断由"对话模式、我卡住了、上下文补词"解决；表达准确性由"候选确认、保守生成、二次确认"保障；长期使用价值由"个人词典、物品记忆、自适应学习"提升。

因此，语桥的核心功能体系应围绕一句话展开：

**通过情境感知、多模态理解和确认式 AI 生成，帮助轻中度失语患者用更少点击完成更准确、更自然、更有尊严的表达。**

## 八、关键算法与技术实现预测

### 8.1 拍照识物算法

可能技术路线：

1. 使用 AVFoundation 调用摄像头；
2. 使用 Apple Vision Framework 进行图像预处理；
3. 使用 Core ML 模型进行常见物体识别；
4. 对复杂场景调用云端多模态模型；
5. 返回多个候选物体；
6. 用户或家属确认正确结果；
7. 系统根据物体名称生成相关表达搭配。

示例：

> 拍照药盒
> ↓
> 识别候选：药盒 / 降压药 / 药瓶
> ↓
> 家属确认：我的降压药
> ↓
> 系统推荐：吃降压药 / 忘了吃药 / 药吃完了 / 帮我拿药

### 8.2 个性化物品匹配算法

可能技术路线：

1. 对用户拍摄的个性化物品提取图像 embedding；
2. 将 embedding 与用户自定义标签绑定；
3. 再次拍照时提取新图片 embedding；
4. 计算相似度；
5. 若相似度超过阈值，返回个性化标签；
6. 若不满足阈值，返回通用识别结果。

核心目标不是实现专业级物体识别，而是让患者常用物品被系统记住。

### 8.3 表达中断检测算法

**第一阶段采用主动触发：**

> 用户点击"我卡住了"
> ↓
> 系统进入补词流程

**第二阶段尝试辅助检测：**

> 开启对话模式
> ↓
> ASR 获取当前语句
> ↓
> 检测停顿时长
> ↓
> 判断句子完整性
> ↓
> 判断是否需要弹出候选

综合判断信号包括：

1. 停顿时长；
2. 当前语句是否未完成；
3. 是否出现"我想……""那个……""帮我……"等开放式结构；
4. 最近对话上下文；
5. 当前时间和地点；
6. 患者历史表达习惯；
7. 是否处于主动开启的对话模式。

系统不应对所有停顿都弹出候选，以避免干扰正常交流。

### 8.4 候选推荐算法

推荐策略可以设计为综合评分：

**候选得分 =**
上下文相关性
\+ 使用频率
\+ 时间段权重
\+ 场景权重
\+ 家属配置权重
\+ 近期表达权重
\+ 视觉识别相关性

示例：

1. 早晨优先推荐"喝水""吃药""早餐"；
2. 晚上优先推荐"休息""翻身""关灯"；
3. 在医院优先推荐"医生""检查""药""疼痛"；
4. 拍到药盒时优先推荐"吃药""忘了吃药""药吃完了"；
5. 点击"我想见"后优先推荐家属照片。

### 8.5 语句整理算法

推荐使用 LLM，但必须加入强约束：

1. 不擅自添加患者未确认的信息；
2. 优先生成简短明确句子；
3. 医疗相关表达保持保守；
4. 歧义较大时生成多个候选；
5. 每次播报前必须确认。

示例 Prompt 约束：

> 请根据患者选择的关键词生成一句简短、清晰、适合日常沟通的中文句子。
> 不要添加患者未明确表达的新信息。
> 如果关键词存在多种可能含义，请生成 2–3 个候选句。
> 医疗相关内容必须保持保守、直接、明确。

## 九、隐私、安全与伦理设计

### 9.1 表达自主权保护

语桥坚持：

**AI 不替患者做决定，患者拥有最终表达权。**

所有 AI 整理后的句子都必须经过患者确认后才能播报。对于含义不明确的关键词组合，系统应提供多个候选句，而不是直接输出唯一答案。

### 9.2 拍照、语音与位置隐私

1. 拍照识别必须由用户主动触发；
2. 对话模式必须显式开启；
3. 默认不进行后台监听；
4. 位置感知必须单独授权；
5. 图像和语音数据优先本地处理；
6. 云端调用前应明确提示；
7. 用户可以删除历史记录；
8. 用户可以关闭个性化学习。

### 9.3 音色复刻安全

音色复刻作为高级可选功能，不作为第一版 MVP 核心能力。

安全原则如下：

1. 默认使用系统 TTS；
2. 音色复刻必须显式授权；
3. 声音样本加密存储；
4. 支持随时删除；
5. 不允许用于冒充他人；
6. 对未成年人、病患、无完全表达能力用户引入监护人授权机制；
7. 医疗、法律等高风险场景保留明显"辅助生成"提示。

### 9.4 医疗边界

本项目不提供医学诊断、治疗建议或康复评估。疼痛、呼吸困难、紧急求助等内容仅作为患者表达辅助，最终处理应由家属或医护人员判断。

## 十、MVP 版本设计

### 10.1 MVP 核心目标

第一版目标不是做全自动对话助手，而是做出一个稳定可演示的辅助表达闭环：

**表达启动 + 主动补词 + 图文化候选 + AI 整理句子 + 二次确认 + 语音播报**

### 10.2 MVP 必做功能

1. 高频表达首页；
2. 拍照识物；
3. 物品相关表达推荐；
4. "我卡住了"主动补词；
5. 3–4 个图文化候选；
6. 句式骨架入口；
7. AI 整理成句；
8. 播报前二次确认；
9. 系统 TTS 播报；
10. 家属添加常用词和照片；
11. 基础表达记录；
12. 隐私开关与数据删除。

### 10.3 MVP 暂不作为核心的功能

1. 全天候自动监听；
2. 完全自动卡顿检测；
3. 眼动识别；
4. 音色克隆；
5. 医疗级康复评估；
6. 复杂多轮自由对话；
7. 多人实时对话分析；
8. 与医院系统直接对接。

这些功能可作为未来拓展。

## 十三、项目创新点

### 13.1 从静态词库到情境感知表达

传统 AAC 通常依赖固定词库，患者需要主动寻找词汇。语桥通过时间、地点、视觉识别、对话上下文和个人习惯学习，主动提供更接近当前情境的表达候选。

### 13.2 从"单词选择"到"表达闭环"

语桥不仅提供词语，还支持"表达启动—候选推荐—患者确认—AI 整理—二次确认—语音播报"的完整闭环，使患者能够从零散词语逐步完成完整表达。

### 13.3 面向失语症的确认式 AI 生成

语桥不让 AI 直接替患者发声，而是采用"候选—确认—整理—确认—播报"的流程，兼顾表达效率与表达自主权。

### 13.4 图文化低负担交互

项目使用图片、图标、真实照片、大字标签和句式骨架，降低患者阅读、打字和组织语言的负担。

### 13.5 个性化表达记忆

系统能够记住患者常用表达、家属关系、个人物品、药品和护理偏好，使推荐内容越来越贴近具体患者。

### 13.6 情感化语音输出

基础版使用系统 TTS，高级版可在授权前提下建立个人声音档案，让患者以更接近自己的声音表达。

## 十四、竞品与差异化分析

现有 AAC 产品和眼控设备已经证明辅助沟通方向具有真实需求，但许多方案存在成本高、配置复杂、中文本土化不足、场景感知弱等问题。

语桥的差异化在于：

| 维度 | 传统 AAC / 通用沟通板 | 语桥 |
|------|----------------------|------|
| 输入方式 | 用户主动找词 | AI 根据情境推荐候选 |
| 词库 | 静态词库 | 个性化词库 + 使用习惯学习 |
| 表达粒度 | 单词或固定短语 | 词、短语、句式、完整句 |
| 表达场景 | 通用表达 | 启动困难 + 表达中断 |
| 场景感知 | 弱 | 时间、地点、视觉、上下文综合感知 |
| 用户确认 | 部分支持 | 全流程确认式生成 |
| 中文适配 | 不一定充分 | 面向中文家庭照护和医患沟通 |
| 语音输出 | 通用 TTS | TTS + 可选个人声音档案 |

语桥的核心竞争力不是单一技术，而是将多模态识别、上下文预测、图文化候选、确认式生成和个性化学习整合为一套失语友好的表达辅助机制。

## 十六、技术风险与应对方案

| 风险 | 说明 | 应对方案 |
|------|------|----------|
| 图像识别错误 | 物体识别可能不准确 | 返回多个候选，让用户确认 |
| AI 猜错含义 | 零散词可能有多种逻辑关系 | 给多个候选句，播报前确认 |
| 卡顿误判 | 正常停顿可能被误认为卡住 | 第一版使用主动触发，自动检测作为拓展 |
| 延迟过高 | 云端模型调用影响对话节奏 | 高频功能端侧完成，复杂生成云端完成 |
| 隐私争议 | 拍照、录音、位置、声音样本敏感 | 主动触发、显式授权、本地优先、可删除 |
| 音色滥用 | 声音克隆可能被误用 | 默认关闭，授权使用，限制场景 |
| 候选过多 | 认知负担增加 | 每次只展示 3–4 个主候选 |
| 患者无法确认 | 严重障碍患者可能无法操作 | 明确目标用户为轻中度患者，后续支持眼控/开关控制 |
| 中文表达复杂 | 中文语义和句式变化丰富 | 使用模板 + LLM 保守生成 |
| 医疗表达风险 | 疼痛、呼吸困难等表达可能影响照护决策 | 设置风险分级和二次确认 |

## 十七、可行性分析

### 17.1 技术可行性

项目所需核心技术均已有成熟路径：

1. 拍照识物可通过 Vision、Core ML 或多模态模型实现；
2. 语音播报可通过 iOS TTS 实现；
3. 候选推荐可通过规则引擎和 LLM 结合实现；
4. 数据存储可使用 Core Data 或 SQLite；
5. 家属照片和个性化词库可通过本地数据库实现；
6. 句子整理可通过大语言模型完成；
7. 表达记录和候选排序可通过本地统计实现。

第一版 MVP 不依赖高难度眼动识别，也不依赖复杂医疗级算法，因此工程落地风险相对可控。

### 17.2 用户可行性

目标用户是轻中度失语患者和命名性失语患者，他们通常仍具备一定理解能力和简单点击能力。项目通过大按钮、图标化候选、真实照片、句式骨架和确认播报降低交互难度，符合目标用户的使用特点。

### 17.3 参赛可行性

项目具备以下参赛优势：

1. 社会价值明确；
2. 用户痛点真实；
3. AI 应用场景具体；
4. 移动端特征明显；
5. 可演示闭环完整；
6. 有创新性，也有落地性；
7. 适合展示"科技向善"和"无障碍设计"；
8. 具备健康医疗、养老照护、社会服务等多重应用价值。

## 二十、总结

"语桥"是一款面向轻中度失语人群的情境感知式 AI 辅助沟通 App。它聚焦命名困难、找词困难、表达启动困难和表达中断困难，通过高频表达、拍照识物、对话补词、图文化候选、句式骨架、AI 语句整理和确认式语音播报，帮助患者以更低负担完成日常表达。

项目的核心不是让 AI 替患者说话，而是让 AI 帮助患者找回想说的话。通过"候选—确认—整理—确认—播报"的机制，语桥既提升了表达效率，也保护了患者的表达自主权。

与传统静态 AAC 沟通板相比，语桥更强调情境感知、多模态理解、个性化学习和中文场景适配。它不仅能帮助患者表达"我要喝水""我疼"等基本需求，也能通过句式骨架和 AI 整理帮助患者表达更复杂、更自然、更有尊严的话语。

项目具有明确的社会价值、较强的技术可行性和良好的参赛展示效果。第一阶段可优先完成"高频表达 + 拍照识物 + 主动补词 + 确认播报"的 MVP 闭环，后续逐步扩展对话模式、个性化学习、家庭声音档案和多模态输入能力。

**语桥希望实现的不只是"把词说出来"，而是帮助失语患者重新获得表达自我、传递需求、维系情感和参与社会交流的能力。**
